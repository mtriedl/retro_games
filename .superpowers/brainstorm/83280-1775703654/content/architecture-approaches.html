Architecture Approaches

How should we structure the game code?

A

Single-File Game (Simplest)

One HTML file, one JS file, one CSS file. Game state is a plain object. Functions organized by concern (rendering, physics, input, AI) in sections within the file.

Pros

  • Dead simple -- open index.html and play
  • Easy to understand top-to-bottom
  • No module system or bundler

Cons

  • One large JS file (~1500-2000 lines)
  • Harder to navigate as it grows
  • All state is global or closure-scoped
B

ES Modules (Recommended)

Separate ES module files: game.js, renderer.js, physics.js, ai.js, audio.js, ui.js. Native browser import/export -- no bundler needed. Each module is 200-400 lines.

Pros

  • Clean separation of concerns
  • Each file is focused and readable
  • Still no build step -- browsers support ES modules natively
  • Easier to work with (both for you and for Claude)

Cons

  • Must serve from a local server (file:// blocks ES module imports)
  • Slightly more files to manage
C

Class-Based ECS (Over-engineered)

Entity-Component-System architecture with classes for entities, a component registry, and system update loops. Full game engine patterns.

Pros

  • Extensible if the game grows significantly
  • Clean abstractions

Cons

  • Massive overkill for a turn-based 2D game
  • Adds complexity without proportional benefit
  • Slower to build

Recommendation: Option B

ES Modules hits the sweet spot -- clean code organization without any build tooling. The "must serve locally" constraint is trivially solved with python3 -m http.server or a one-line script. Each module stays small enough to reason about independently, and the game's natural boundaries (rendering, physics, audio, AI, UI) map directly to files.
